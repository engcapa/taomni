# Code Workspace 性能优化待办

> 日期：2026-07-13  
> 状态：待实施  
> 关联修复：`934e7ad fix(code-workspace): stop command registration update loop`

## 1. 背景与当前结论

此前在 Windows 11 和 Linux 上打开 Code Workspace 后出现 CPU 持续升高，并可能最终触发 React error #185（Maximum update depth exceeded）。根因是命令注册链路中的对象和回调引用不稳定，形成了子组件发布命令、父组件更新状态、子组件重新渲染并再次发布命令的反馈循环。

提交 `934e7ad` 已切断该循环，并补充了稳定引用、注册生命周期和父子状态反馈的回归测试。修复版本在 Windows 11 和 Linux 的实际打包测试中表现为：

- CPU 占用保持较低；
- 暂未再次出现 React #185 崩溃；
- 空 Code Workspace 不再产生明显的持续 CPU 消耗。

因此，原来的持续高 CPU 与崩溃可以认为是同一主问题的不同表现。下面的事项不是该崩溃修复的必要组成部分，而是针对大型工作区、后台任务和频繁编辑场景的进一步性能收口。应在独立性能分支中实施，避免扩大崩溃热修复的变更范围。

## 2. 优化目标

- 空工作区打开并稳定后，不产生持续渲染、轮询或状态写入。
- 非当前可见的 Code Workspace 尽量停止 Git、LSP、diff 等后台工作。
- 打开大型工作区时，首屏优先，不因恢复大量文件而长时间占用 CPU、磁盘或 UI 线程。
- 编辑大文件时，减少全文字符串复制、无效 React 重渲染和 CodeMirror 重配置。
- 所有优化必须保持文件恢复、Git 状态、LSP、诊断、状态栏和编辑器命令行为不变。

## 3. P0：低风险、优先实施

### 3.1 收敛状态栏更新

- [ ] 将 Code Workspace 状态栏的“内容更新”和“卸载清理”拆成独立 effect，避免每次依赖变化时先清空再写回。
- [ ] 稳定状态栏 action 回调及 action 对象，避免仅因对象引用变化写入 Zustand。
- [ ] 为状态栏数据增加相等性判断；内容未变化时不更新 store。
- [ ] 检查状态栏 effect 是否不必要地依赖完整编辑器文本；光标或语言信息不应由全文变化驱动。
- [ ] 增加测试，验证光标未变化、状态未变化及无关父组件重渲染时不会重复发布状态栏数据。

预期收益：减少输入、切换光标和无关重渲染时的 store 写入及跨组件刷新。

### 3.2 消除不稳定的空数组和无效 CodeMirror 重配置

- [ ] 为 diagnostics、Git changes、highlights、inlay hints、semantic tokens、breadcrumbs 等空值使用模块级稳定常量，避免在 render 中生成新的 `[]`。
- [ ] 在 CodeMirror compartment 更新前增加语义相等判断；配置内容未变化时不执行 reconfigure。
- [ ] 对光标、视口和空结果回写增加相等性判断，避免相同值重复进入 React/Zustand 状态。
- [ ] 增加回归测试，确认无关父组件重渲染不会重复配置 diagnostics 和 Git 装饰。

预期收益：降低编辑器空闲和父组件重渲染时的扩展重配置成本。

### 3.3 限制 Git snapshot 后台轮询

- [ ] `useWorkspaceGitSnapshots` 仅对当前可见的 Code Workspace 进行定时轮询。
- [ ] 工作区重新变为可见时立即刷新一次，随后恢复正常轮询周期。
- [ ] 增加同仓库共享缓存，避免多个工作区或 tab 对同一仓库重复请求。
- [ ] 增加 in-flight 防重入，上一轮未完成时不启动下一轮。
- [ ] 对 snapshot 结果做语义比较；结果未变化时不写入 loading/result 状态。
- [ ] 工作区隐藏或卸载时可靠取消 timer，并忽略已经过期的异步结果。
- [ ] 增加 fake timer 测试，覆盖可见、隐藏、重新激活、重复仓库和慢请求场景。

预期收益：减少大型 Git 仓库、多个工作区以及 Windows Defender 参与文件访问时的周期性 CPU 和磁盘峰值。

### 3.4 缓存并延迟 LSP 服务探测

- [ ] 将 `lspDetectServers()` 的结果做应用级缓存，并设置合理的失效或手动刷新机制。
- [ ] 空工作区且没有受支持文件时，不立即探测全部语言服务器。
- [ ] 优先在打开受支持文件、展开 LSP 面板或用户主动刷新时触发探测。
- [ ] 后端缓存命令可用性/PATH 查询结果，避免对多个工作区重复执行 `which` 类扫描。
- [ ] 并发调用共享同一个探测 Promise，防止启动阶段重复扫描。
- [ ] 增加测试，验证多个工作区只触发一次探测，且手动刷新能够更新缓存。

预期收益：减少工作区首次打开时的短时 CPU 峰值，Windows 上收益预计更明显。

## 4. P1：大型工作区优化

### 4.1 工作区文件恢复改为首屏优先

- [ ] 恢复布局时只优先读取每个 editor group 的活动文件，其他 tab 先恢复元数据占位。
- [ ] 用户切换到未加载 tab 时再读取对应文件内容。
- [ ] 必须预加载时使用 2～4 的受控并发，不同时启动全部文件读取。
- [ ] 批量更新 loading 状态，避免每个文件分别触发整棵工作区组件树重渲染。
- [ ] 保持原活动 tab、分组顺序、dirty 状态和 loose file 行为不变。
- [ ] 增加包含多 editor group、多文件、读取失败和快速切换工作区的恢复测试。

预期收益：降低启动/恢复包含几十个编辑器 tab 的工作区时的 CPU、磁盘并发和首屏延迟。

### 4.2 Git 行级 diff 仅按需计算

- [ ] 默认只为当前可见编辑器文件计算行级 diff。
- [ ] 以文件路径、HEAD 内容版本和工作区文本版本作为缓存键，复用未变化结果。
- [ ] 文件切出可见区域后取消或忽略过期计算。
- [ ] 对大文件设置阈值和降级策略；必要时将 diff 计算移动到 Web Worker。
- [ ] 避免对 `gitHeadTextByFile` 中所有文件进行全量遍历和延迟调度。
- [ ] 增加快速输入、快速切换文件、HEAD 变化和大文件降级测试。

预期收益：减少多文件打开和频繁输入时的 diff CPU 消耗及 UI 线程阻塞。

### 4.3 缩小 React 重渲染范围

- [ ] 将 `CodeWorkspaceTab` 对 workspace store 的整实例订阅拆成细粒度 selector，并使用浅比较或语义比较。
- [ ] 对 ProjectTree、EditorGroup、BottomDock 等高成本区域增加明确、可测的渲染边界。
- [ ] 非活动的 Code Workspace tab 保留必要状态，但暂停不必要的计算和 effect。
- [ ] BottomDock 中无需保持实例状态的隐藏面板改为按需挂载；必须保留状态的面板暂停数据刷新。
- [ ] 使用 React Profiler 或现有 probe 的开发版变体确认优化前后的提交次数；不要把高频 probe 埋点带入生产构建。

预期收益：避免一次局部状态更新导致项目树、两个编辑器分组和所有底部面板共同重渲染。

### 4.4 文件树规模化处理

- [ ] tree/compact/flat 模式共享并缓存目录排序键和扁平化结果。
- [ ] 仅在根目录、过滤词或树节点数据变化时重建过滤索引。
- [ ] 大量节点使用虚拟列表，避免一次创建数千个 DOM 行。
- [ ] Git 嵌套仓库发现结果按根目录和文件系统版本缓存，避免重复深度扫描。
- [ ] 保留现有深度、节点数安全上限，并在截断时给出可理解的 UI 提示。
- [ ] 增加 2,000+ 节点、深层目录、快速过滤及多根目录测试。

预期收益：改善大型仓库首次展开、flat 模式和文件名过滤时的响应速度。

## 5. P2：编辑器数据流长期优化

当前编辑过程中，CodeMirror 文档可能被频繁转换为完整字符串并写回 React/Zustand，随后又以完整字符串与 CodeMirror 当前文档比较。大文件输入时，这会增加 O(n) 字符串复制、父组件重渲染和 LSP/Git 下游计算。

- [ ] 让 CodeMirror 成为活动编辑器实时 buffer 的权威数据源，React store 主要保存元数据、版本号和持久化快照。
- [ ] 保存、切换、后台分析等边界事件再获取完整文本；普通按键不立即把全文写入顶层 store。
- [ ] LSP 优先发送增量 `contentChanges`；不支持增量同步的 server 再使用节流后的全文同步。
- [ ] Git diff、语义高亮等消费者基于文档版本调度，并对中间版本进行合并或取消。
- [ ] 明确 dirty、自动保存、外部文件变化、undo/redo、分屏共享同一文件等一致性规则。
- [ ] 在实施前补充单独设计文档和迁移测试；此项不与 P0/P1 混在同一提交中。

预期收益：从根本上改善大文件连续输入时的 CPU、内存分配和渲染延迟。

## 6. 测量与验收场景

优化前先记录基线，优化后使用相同打包方式、相同测试仓库和相同观察窗口复测。至少覆盖 Windows 11 与 Linux。

### 6.1 必测场景

- [ ] 真正空工作区：无 roots、无 loose files，打开后静置 10 分钟。
- [ ] 空目录根工作区：有 root，但目录内无代码文件，静置并跨越至少两个 Git 轮询周期。
- [ ] 大型 Git 仓库：2,000+ 文件，观察初次打开和每 30 秒附近是否出现周期峰值。
- [ ] 恢复型工作区：两个 editor group、20～40 个已打开文件，记录可交互时间和峰值。
- [ ] 大文件编辑：分别使用 1 MB、5 MB 文本，连续输入、撤销、切换文件。
- [ ] 多工作区切换：打开 3 个 Code Workspace，反复切换可见 tab，确认隐藏 tab 不持续工作。
- [ ] 文件树压力：tree/flat 模式切换、过滤输入、批量展开目录。
- [ ] LSP 场景：未安装语言服务器、已安装多个语言服务器、手动刷新探测。

### 6.2 建议记录指标

- 空闲稳定后的平均 CPU 与短时峰值；
- WebView2 Renderer/网页渲染进程与 Tauri 主进程分别占用；
- 磁盘读取量与周期性访问；
- 打开工作区到首个活动文件可编辑的时间；
- React commit 次数和关键组件 render 次数；
- CodeMirror reconfigure 次数；
- Git snapshot、LSP detect、文件读取和 diff 的调用次数及耗时；
- 大文件连续输入时的长任务数量和输入延迟；
- 稳定后的内存占用，以及多次打开/关闭工作区后是否持续增长。

建议把“相对基线变化”作为主要判断依据，不先写死跨机器不可靠的绝对 CPU 百分比。空闲状态的核心门槛是：没有持续反馈循环、没有无意义状态写入，周期任务的调用次数与设计一致。

## 7. 实施顺序与提交边界

建议建立独立分支，例如 `perf/code-workspace-idle-and-large-files`，按以下顺序实施：

1. 状态栏 effect 拆分和稳定空值；
2. Git snapshot 可见性、去重和结果相等判断；
3. LSP 探测缓存与延迟触发；
4. 工作区恢复懒加载和受控并发；
5. 活动文件按需 diff；
6. 组件订阅拆分、文件树虚拟化；
7. 单独设计并实施编辑器 buffer 数据流重构。

每一项使用独立、可回滚的提交，并附对应单元测试与优化前后数据。不要将功能开发、视觉调整或无关重构混入性能提交。
